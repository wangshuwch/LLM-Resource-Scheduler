# LLM资源调度器 - 阶段二实现规格说明（修订版）

## Overview
- **Summary**: 基于设计文档重新实现阶段二功能，包括完善场景配置模块、优化调度引擎的资源分配逻辑、完善请求处理流程，并解决系统中遗留的问题。
- **Purpose**: 确保系统在资源充足场景下能够充分利用资源，同时支持场景配置的热更新，为后续的资源竞争场景实现做好准备。
- **Target Users**: 系统开发人员和运维人员。

## Goals
- 完善场景配置模块，支持配置热更新
- 优化调度引擎的资源分配逻辑，确保资源充足场景下的高效分配
- 完善请求处理流程，包括状态跟踪和结果查询
- 解决系统中遗留的问题，确保系统稳定运行
- 编写完整的单元测试和集成测试

## Non-Goals (Out of Scope)
- 实现资源竞争场景下的优先级分配算法（阶段三任务）
- 实现请求队列管理和超卖机制（阶段四任务）
- 实现系统集成测试和性能优化（阶段五任务）

## Background & Context
- 阶段一已经完成了基础架构搭建和资源监控模块的实现
- 现在需要在阶段一的基础上，实现资源充足场景下的分配机制
- 系统需要支持场景配置的热更新，确保配置变更能够实时生效

## Functional Requirements
- **FR-1**: 场景配置模块增强
  - 支持场景配置的创建、更新、删除和查询
  - 支持配置的热更新，无需重启服务
  - 支持配置参数的合法性验证
- **FR-2**: 资源充足场景分配逻辑
  - 当总请求 ≤ 总负载时，各场景应能按配置的最大资源限制充分使用资源
  - 资源充足时请求直接处理，无需排队
  - 某场景请求量超过该场景最大资源限制时，该场景的额外请求应进入队列等待
- **FR-3**: 请求处理流程完善
  - 支持请求状态的完整跟踪（PENDING → PROCESSING → COMPLETED/FAILED）
  - 支持请求结果的存储和查询
  - 支持Token消耗统计

## Non-Functional Requirements
- **NFR-1**: 性能要求
  - 配置更新应在100ms内生效
  - 资源充足场景下，请求处理延迟应低于50ms
- **NFR-2**: 可靠性要求
  - 系统应支持并发请求处理
  - 配置更新应保证线程安全
- **NFR-3**: 可测试性
  - 所有功能应提供完整的单元测试
  - 关键功能应提供集成测试

## Constraints
- **Technical**: Python 3.11+, FastAPI, Redis（队列存储）
- **Dependencies**: tiktoken（Token计算）

## Assumptions
- 系统运行在单节点环境下
- Mock LLM资源池能够模拟真实LLM服务的行为
- 配置变更频率较低，不会频繁触发热更新

## Acceptance Criteria

### AC-1: 场景配置管理
- **Given**: 系统已启动，场景配置管理器已初始化
- **When**: 用户创建或更新场景配置
- **Then**: 系统应验证配置参数的合法性
- **Then**: 配置应实时生效，无需重启服务
- **Then**: 系统应支持查询所有场景配置
- **Verification**: `programmatic`

### AC-2: 资源充足场景分配
- **Given**: 系统总负载 ≤ 总资源限制
- **When**: 各场景提交请求
- **Then**: 各场景按自身最大资源限制内的请求应直接处理，无需排队
- **Then**: 资源利用率应最大化
- **Verification**: `programmatic`

### AC-3: 场景资源限制
- **Given**: 某场景请求量超过该场景最大资源限制
- **When**: 该场景继续提交请求
- **Then**: 该场景的额外请求应进入队列等待
- **Then**: 其他场景仍可正常使用资源
- **Verification**: `programmatic`

### AC-4: 完整请求处理流程
- **Given**: 系统已启动，请求已提交
- **When**: 请求被处理
- **Then**: 请求状态应从 PENDING → PROCESSING → COMPLETED/FAILED
- **Then**: 请求结果应记录Token消耗统计
- **Then**: 可查询请求状态和结果
- **Verification**: `programmatic`
